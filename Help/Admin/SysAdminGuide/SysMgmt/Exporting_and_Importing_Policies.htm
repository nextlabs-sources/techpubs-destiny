Exporting and Importing Policies

Components and policies require time and effort to design and construct. How­ever, there may be cases where you want to distribute the components and pol­icies defined in one implementation of Control Center and deploy them in another. For example, you might have a separate test environment where you have fine-tuned your policies, and you now need to transfer them to your pro­duction environment.

To avoid having to manually redefine all the components and policies in the sec­ond system, Control Center provides an import/export feature. This feature is available both in the Policy Manager user interface in Policy Studio, and through a command-line import/export utility. It allows you to export any policies in one implementation, along with their folder structure, so you can import them else­where. Whenever you export policies, all the components they require are auto­matically included. All objects are exported to an XML file, which you can then import into the policy repository of another Control Center system.

Specifically, the feature allows you to:

·        Export all policies from a given Control Center

·        Export selected policies by folder, recursively

·        Export selected individual policies

·        Perform a shallow import, which strips policies of any leaf wirings that tie them to the physical system

·        Display a list of policies and components, without exporting

·        Import the contents of XML files generated by exporting

·        View the contents of such XML files, without importing

Again, you can use this feature either directly in Policy Studio, or through the pair of utilities. The following sections describe the utilities. For information about exporting and importing policies using Policy Manager, see Policy Studio User’s Guide.

Exporting

To export an existing set of policies and/or components:

  1. Log in to the host of the Control Center from which you want to export policies, or to a computer that can connect to it, and open a command line window.

  2. Make sure the Export utility (export.bat) is present on the host where you will be working. By default, during installation it is placed on the Control Center host at the following location:

Export Target Specifications

-a

Export file will include all folders, policies and their dependent components in the Policy Master database. Supersedes any other target specification.

-d <dirName>

Include specified directory, exporting all policies and their dependent components, but not including any subdirectories. May be used more than once, and in conjunction with any other target specifier.

-r <dirName>

Include specified directory, including all its subdirectories, exporting all policies and their dependent components. May be used more than once, and in conjunction with any other target specifier.

-P <path/policy>

Include the policy specified by path/policy, including all of its dependent components. The path is the folder hierarchy of this policy. May be used more than once, and in conjunction with any other target specifier. Note that this must be a capital “P”.

-F <filename>

The name you want to assign to the export file. The value specified here will be appended with an .xml extension. This argument is required for all commands except List (-l), and must be the last string in the line (space-delimited). Note that this must be a capital “F”.

Conflict Handling

You have two choices for how the export utility behaves if it encounters an existing file with the same name as specified by the -F parameter.

·        Create the new export file and rename the existing one by appending a date/time stamp (-f)

·        Cancel the entire export operation (-x)

If neither option is specified in the command line, the utility explicitly prompts you with a standard “Overwrite Y/N?” message whenever it encounters a con­flict.

Displaying a List

If you only want to generate a list of policies and/or components, you can use the -l argument, which returns a list of all folders, policies and components specified by the export target specification elements, but without the details—just the names.

Note: You must specify the target policies or folders you want to list.

Importing

Once you have generated an XML file through the export procedure, you can import it to another Control Center, where the policies and components are then available for use in Policy Studio.

You use a separate import utility called import.bat, which is also placed in the Program Files\NextLabs\PolicyServer\tools\importexport directory dur­ing Control Center installation.

To use this utility:

  1. Make sure no Policy Studio users anywhere in the system are currently working on any of the policies you want to import (see Open Policies).

  2. Log in to the host of the Control Center from which you want to export policies, or to a computer that can connect to it, and open a command line window.

  3. Make sure the Import utility (import.bat) is present on the host where you are working. During installation it is placed, by default, in the Control Center host in Program Files\NextLabs\PolicyServer\tools\import­export.

  4. Navigate to the location of the utility, and type the import command line, specifying the required connection arguments, controls, and name of the source file you want to import. Use the following format:

    5.  

    Note: On Duplicate Objects: Although they are technically optional, we strongly recommend that you specify one of the conflict-handling con­trols (-o, -r, -d, or -x), in order to specify how the utility should behave if the import file contains objects that are already defined in the tar­get system. There are nearly always duplicated objects between one system and another—if nothing else, the seed data that is typically imported for use in policies is often duplicated in all systems.

    The conflict-handling controls are mutually exclusive—you can only spec­ify one of the four. If you do not include any of them in your command line, the import proceeds, but all duplicate objects are imported into the new object library.

    For definitions of these controls, see the table below.

  5. When the import finishes, it displays one of the following return codes:

0: Import successful
-1: Unknown host
-2: Connection refused. Check the credentials and try again.
-3: Cannot open export file. File does not exist, or user has inadequate permissions. Import aborted.
-4: Import canceled by user.

About Shallow Import

The shallow import flag, -i, commands the utility to perform a special kind of import: for any components or policies that are defined with reference to spe­cific user or group names or to network hosts or paths, those references will be removed before importing. Only aspects of the definitions that do not tie the definitions to the physical environment—such as LDIF metadata and file or user properties—are included in the import. The original policy tree folder structure, and nested component structures (components whose definitions contain other components) are also preserved.

Conflict Handling

You have four choices for how the import utility behaves if it encounters a name conflict between imported components and policies, and those already defined on the target system.

·        Overwrite existing data (-o)

·        Import new data, adding a date/timestamp to each name (-r)

·        Keep existing data and ignore imported data (-d)

·        Cancel the entire import operation (-x)

Because these behaviors are mutually exclusive, you can specify only one of the four flags for each import operation. If you do not specify any of them, the -r behavior executes by default.

Viewing the Contents of an Export File

As with the Export command, the -l flag allows you to review the contents of the file before you actually run an import procedure.

Permissions for Imported Policies

Policies and components that are imported are assigned the permissions defined for the Default Access Group (DAC) associated with the importing user. Users who are members of the group associated with the importing user also have access to the imported policies and components. If the DAC group for the importing user does not have adequate permissions to access policies, other users in that group also do not have adequate permissions.

Open Policies

If any policy analyst is using Policy Studio and has a policy or component open at the time a newer version is imported from a file, that object is not updated. This should be avoided, since it produces an inconsistent policy set. System administrators can avoid this in the following ways:

·        Perform the update at a time when you are certain no one is using any instance of Policy Studio.

·        Request that all policy analysts shut down their Policy Studio while any policies are being updated by importing.

·        Publish the list of policies and components planned for updating by import, and ask all policy analysts to refrain from editing any object in that list until the update is complete.

Policy Studio Restart

Note that if the target Policy Studio—the one installed on the system where you are importing policies—is running, you must restart it before the imported poli­cies and components are available in the policy tree and component panels.
