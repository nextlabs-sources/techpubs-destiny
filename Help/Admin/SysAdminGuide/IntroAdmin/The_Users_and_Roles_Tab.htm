The Users and Roles Tab

The controls available on this tab allow you to define and manage users, user groups, and roles for the Control Center tools—Policy Studio, Administrator, and Reporter.

 

Users And Roles tab and sub-tabs

Note: The term users in this context is distinct from the two default user accounts that are available when you initially install Control Center: one for logging in to the platform’s software tools, and one for logging in to the Information Network Directory’s underlying data server. These accounts are not managed with the Administrator tool, and, therefore, are not covered in this section.

About Users

In the Administrator tool, the term user refers to people who need to use the Control Center tools—Policy Studio, Administrator, and Reporter. For example, users include policy designers, Control Center administrators, and executives or auditors running reports. Each user has a login name and password, which is used to verify the identity of the person attempting to use a Control Center tool.

These users can be defined in or imported into Control Center, which stores them in the Information Network Directory. The stored user information can come from the following sources:

·        Imported into Control Center from an organization’s Active Directory. You can use this technique to grant existing Windows users access to Control Center applications.

·        Manually added as a new user.

Both types of user can be set up as Control Center users through the Users and Roles tab in Administrator.

For information about how users are authenticated, and how to authenticate them locally (within Control Center) or using a live connection to the organiza­tion’s information network, see Configuring Authentication.

About Groups

A group is a set of users who have similar responsibilities and require similar rights to Control Center objects, such as policies, policy components, and reports. Rather than setting these rights for each user individually, you can set rights for the group and then assign users to the appropriate group.

Similar to users, groups can be manually defined in Administrator or imported from an organization’s Active Directory. For example, if you already have an LDAP group called System Administrators, you can leverage this group as a source of Control Center users. The benefit of linking Control Center groups to existing Windows groups is that if the group membership changes in Windows, Control Center automatically picks up the changes. You can continue to manage the group on the Windows side. If you create a new group in Administrator, it is created strictly within Control Center and does not have any relation to the LDAP groups imported from Active Directory.

About Roles

In Control Center, a role refers to a set of permissions that determine which tasks individual users assigned to that role can do with Control Center. Permis­sions within each role include which Control Center applications can be used and what permission is granted for creating policies and components in Policy Studio.

Control Center provides the following pre-defined roles: System Administrator, Policy Administrator, Policy Analyst, Reporter Administrator, and Business Ana­lyst. The roles are typically used as follows:

·        System Administrator: System administrators or other IT professionals who need to use Administrator to monitor the Control Center system and manage users and policy enforcer profiles. These users might also need to use Policy Studio and Reporter, and have permissions to create policies and reports. In addition, they are the only users who would have any need to use the Enrollment Manager utility.

·        Policy Administrator: These users typically have responsibility for manag­ing policies and policy components, so they would need access to Policy Studio, with permissions to create and deploy those objects. Typically, these users also require access to Reporter to monitor and audit the effec­tiveness of deployed policies.

·        Policy Analyst: Users with access to Policy Studio in order to construct policies and submit them for deployment. These users might also need to use Reporter to see how their policies are working out in practice.

·        Report Administrator: Users who will use Reporter to create reports on policy enforcement, and make the reports available to other users. A Report Administrator can also create policy monitors and alerts to obtain notifications of critical policy enforcement activities when specified crite­ria or thresholds are met.

·        Business Analyst: Managers, executives, or other security personnel who will audit how policies are operating in an enterprise. Typically, these users only need to use Reporter to view reports on policy enforcement activities and trends.

These typical responsibilities are reflected in the default permissions assigned to the roles. You can leave these default settings as they are, or you can modify them to suit your needs. The exception to this is Enrollment Manager permis­sion, which is available only to System Administrators. By default, users assigned the roles that permit creating policies or policy com­ponents have full control over the policy objects they create, that is, users can edit, delete, submit, and deploy their own policy objects, as well as, grant other users access to their policy objects. You can, however, change a user’s permissions to her policy objects. You do so in the Users tab, as described in the next section.

Managing Users

Managing users involves two basic procedures: adding and defining new users, and deleting existing ones.

Adding a New User

You can use Administrator to create new Control Center user accounts from two sources:

·        Users enrolled from your organization’s Active Directory and authenti­cated using integrated Windows authentication.

·        New users added manually to Control Center. These accounts are in addi­tion to those imported from your organization’s Active Directory and are kept in a separate, local domain. This is useful when, for example, you want to provide a unique login name and password that can be used to log in to Policy Studio (or another Control Center tool). Such accounts are always authenticated locally, even if Control Center as a whole has been configured for remote authentication (see Configuring Authentication).

To set up a new Control Center user account:

  1. Click the Users and Roles tab, then click Users.

  2. Click one of the following buttons:

    3. ·        New to create a new user within Control Center

    ·        Add Windows User to set up an account for a user enrolled from your Active Directory

  3. In the editor pane, specify the properties of the user account. Some of the following fields are not available if you clicked Add Windows User, because in that case, user authentication is set up outside of Control Center.

    4. ·        In User ID, type the user account name that the user types in the Login dialog box when starting Administrator, Reporter, or Policy Studio.

    ·        In First Name and Last Name, type the user’s real name.

    ·        In Password and Confirm Password, type the user’s login password. The password must contain 7-12 characters, at least one number, at least one letter, and at least one non-alphanumeric character other than under­score.

  4. Under Permissions and Rights, in Assigned Roles, check one or more boxes to specify what role or roles to assign to the user.     6. ·        Select a default access control (DAC) group to assign to this user. The drop-down list displays all the groups to which this user belongs. When you assign a DAC group to a user, you are essentially apply­ing the access permissions (for example, Read, Write, Delete, Submit, Deploy policy objects) that are specified for the selected group. In addi­tion, any policy object—folders, components, and policies—that this user creates is automatically available to all users in the selected group.

    ·        Use the default option, None Selected, if you don’t want to apply any group-specified access permissions to this user. The user will have full access to all the policy objects she creates, and, by default, any policy object that this user creates can be accessed by this user only. The user, however, can grant other users access to her policy objects later by chang­ing the objects’ access properties in Policy Studio.

    If you want to allow this user to import seed data and make it accessible to all users, select All Policy Server Users from the list. You must, how­ever, first add this user to the All Policy Server Users group, as described in Managing User Groups. For information about seed data, see Importing Seed Data.

    For more information about how DAC groups work, see About Default Access Control Groups.

  6. Click Save. Once the new user is saved, a confirmation message appears.

Deleting a User

You might need to delete a user if that person leaves your organization or is reassigned to another project and is no longer working with Control Center. When you delete a user, it only affects his or her ability to use Control Center applications; the user is not deleted from your organization.

To delete an existing user, select it from the Users list in the Users screen, and click the Delete button.

Managing User Groups

Managing user groups involves these operations: creating new groups, importing existing groups from your enterprise, and modifying existing groups in Adminis­trator.

Administrator provides a predefined group: All Policy Server Users. As the group’s name suggests, you can use this group to assign basic permissions that you want all users to have when working with policies and policy components. You must assign users to this group to grant them permission to use seed data (pre-defined policy components) that is imported into Policy Studio. This is a typical administrative task.

About Default Access Control Groups

Once you create a group, you can assign the permissions you want other people to have on components and policies created by any user in that group. Assigning default access control (or DAC) at the group level makes it easy for different teams within your organization to work together, or to keep their work separate, as needed. This feature lets you selectively grant access to other users. You can grant different permissions to user groups and to individual users.

You define permissions for access groups in the Default Access Control tab of the Groups screen, as shown in the following figure.

 

The Groups screen, Default Access Control tab

You have the option to allow other users or groups to do any of the following to components and policies created by users in this group:

·        Set Access

·        Read

·        Write

·        Delete

·        Submit

·        Deploy

Although these permissions are set at the group level, they only have meaning insofar as they apply to all the members of that group. After all, groups don’t define components or policies, individual users do. Membership in a group is ultimately determined by the Default Access Control Group field in the Users screen (see The Users screen), which you use to associate each user with a DAC group.

There are two reasons this explicit association in the Users screen is required even though a user may already be assigned to a standard user group in the Groups screen. First, a user may belong to more than one user group with differ­ent DAC permissions defined for them, and Default Access Control Group field allows you to designate which permissions apply to this user. Each user can belong to only one DAC group (even though he may belong to several user groups).

Second, you have the option of not assigning a user to any DAC group by simply leaving the Default Access Control Group field blank. This allows an override mechanism—even if a user belongs to a user group and that group has DAC per­missions defined for it, that user’s components and policies are not subject to those permissions. In other words, the user still has full permissions to his own components and policies. In addition, only the user (and the Control Center administrator) can access those components and policies.

As mentioned previously, DAC permissions can be granted to groups or individu­als. The groups given permission may include the group itself—in which case all members of that group are given the specified permission to any objects others in the group create—but this is not necessary. In the example shown above, the Product Managers group permissions list includes the Product Managers group itself. The example defines the following situation:

·        Whenever any user in the Product Managers group creates a component or a policy, other users in that group have permission to read, write, delete, and submit it. Note that for the component or policy to be automatically shared with users in the group, the owner of the component or policy must also have her Default Access Control Group property set to Product Managers.

·        The user Andy Han has permission to read the components or policies cre­ated by any user in the Product Manager group, even though he is not a member of that group.

·        Users in the Policy Admins group can deploy components and policies cre­ated by users in the Product Manager group.

This example illustrates how to segregate the responsibilities of creating and deploying policies and policy components. Product Managers are permitted to create policies, but not deploy them. A policy administrator is authorized to deploy those policies and components, but cannot read or delete them. You might want to do this if there are multiple users or groups creating policies, and you need a policy administrator to manage which policies are deployed, and when. Segregating areas of responsibility also reduces the opportunity for unau­thorized or unintentional changes to policies.

One last point: the reason these are called default access controls is that they automatically apply when a component or policy is created, but they can be overridden—if the user has Set Access permission—by defining a custom set of permissions for the component or policy in Policy Studio, using the Access Con­trol tab of the object’s Properties window. To assign default access controls to the policies and components created by users of a group:

  1. On the Groups screen, click the Default Access Control tab. Click Add. An index of users and groups appears.

  2. Click the hyperlinked letters to find the name of a user or group to which you want to grant access, then click the name. The name appears in the Principals to Add list at the left side of the window.

  3. When all the desired names are in the list, click Add.

  4. Check one or more boxes next to each principal to indicate what you want that user (or users in that group) to be able to do with objects created by users in the group you are editing.

Linking to an Existing Windows Group

  1. On the Groups screen, click Add Windows Group.

  2. Specify the group(s) you want to link to. An alphabetized index of hyper­links appears, showing groups that exist in your Windows domain.

  3. Click the hyperlinked letters to find the name you want, then click the name. The name appears in the list at the left side of the window.

  4. When all the desired names are in the list, click Add.

  5. On the General tab, give a name and description for the group. The source of the group is displayed in External Link.

Modifying a Group

If the link icon appears, you cannot modify the membership of a group; it is linked to a group in your Windows domain and is managed there, not from within Control Center.

  1. Click the Groups tab. If this group is imported from your Windows domain, its source is displayed in External Link.

  2. Make the desired modifications.

  3. Click Save.

Deleting a Group

To delete a group, select it in the list of groups, and click the Delete button.

Managing Roles

As described earlier, a role is simply a named set of permissions for working with the Control Center applications. You can define roles independently and then assign to users as appropriate. You can use roles as intensively as you wish to suit your needs. However, there are certain circumstances when you should modify them:

Initial Implementation. When Control Center is first installed, the role defini­tions are not assigned to any users. Before any system administrators in your organization can start using the Policy Studio, Reporter, and Administrator tools, you must set up the Control Center roles and assign each role to at least one user. To gain initial access to the Administrator tool in order to set up and assign roles, use the built-in Administrator account.

Shifting Responsibilities. Change the roles to set up the division of labor that reflects the changing conditions at your organization. The permissions granted in each role represent the tasks that can be done by users assigned to that role. In any given organization, at one point in time, the different tasks that can be accomplished with the Control Center tools (Policy Studio, Administrator, and Reporter) might be carried out by different people, while in another organiza­tion or at a different time, all of these tasks might be carried out by one person.

When you modify a role, your changes are not automatically saved as you go. If you are interrupted while working on a role, or if you want to work on another task and return to modifying the role later, click Save before you stop work.

To modify a role:

  1. Click the Users and Roles tab, then click the Roles link.

  2. Click the name of the role you want to modify, to display the current set­tings for that role.

  3. In Applications, select the Control Center tools that you want users in this role to be able to run. For example, for the Business Analyst role, you would typically select Reporter Analysis only.

  4. In Policy Edit, select the types of policy objects that users in this role can create. You can make selections in this section only if you selected Policy Author in step 3. If you select Policy Author, but don’t check any objects in this step, then users in this role can view, but not create policies or pol­icy components.

    5. As described earlier, by default, users that are permitted to create poli­cies or policy components have full control over the objects they create. They cannot access objects created by other users. To change any of these permissions or access rights, you set a user’s Default Access Control in the Users and Groups tabs.

  5. Click Save.

Assigning Roles to Users

Once you have defined users, groups, and, optionally, redefined roles, you need to give each user appropriate access to the Control Center tools (Policy Studio, Administrator, and Reporter). You do this by assigning one or more roles to each user. To do this:

  1. Click the Users and Roles tab, then click the Users link.

  2. In the Users list, click the name of the user to whom you want to assign one or more roles. (The first time you use Administrator, this list is empty.)

    3. If you do not see the user you want, create the user, as described in Adding a New User.

  3. In Permissions and Rights, in Assigned Roles, check one or more boxes next to the roles you want this user to have.

  4. Click Save.
