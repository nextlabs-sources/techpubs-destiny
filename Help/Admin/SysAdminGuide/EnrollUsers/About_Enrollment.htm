About Enrollment

In Control Center, enrollment refers to the process of importing information about your environment into the Control Center system so it is available for use by Control Center components. This can include LDAP directory entities, Share­Point groups, and other kinds of information. When you perform an enrollment, information about the entities in your system is copied and stored in the Infor­mation Network Directory, one of Control Center’s internal databases.

After enrollment is complete, the enrolled data is available for use in Policy Author. For example, you can write policies that restrict the document usage permissions of certain users (or whatever type of enrolled information).

Enrollment occurs as two basic steps: enrolling and syncing. Enrolling refers to mapping data types to the Control Center in preparation for that data actually being populated. Syncing refers to actually retrieving data values and populat­ing the Information Network Directory.

About the Information Network Directory

The Information Network Directory (IND) is the database schema where Control Center stores its information about the entities in your organization—users, computers, groups, applications, and so on. This data is organized according to a structured information resource model, which provides consistent definition and organization to make it possible to define policy components. The Informa­tion Network Directory is installed on the same database host as the Activity Journal, in PostgreSQL, Oracle, or SQL Server.

When you enroll information, you establish a connection with an enrollment domain. In this context, the term “domain” refers to a source where enrollment information resides, not a network domain. In other words, you can have multi­ple enrollment domains to enroll from different sources within a single Active Directory in a single network domain. You can enroll entities from any number of enrollment domains.

You should perform your initial enrollments right after you install Control Cen­ter. As your organization’s infrastructure changes over time, you can manually update the information whenever you need to, and you can also configure the Enrollment Manager to automatically synchronize the information resource model at regular intervals.

What Can Be Enrolled

The Control Center enrollment utilities import information about the underlying information network, including the entities listed below. Note that most are mandatory, and will have to be imported in all installations. Only location sites and SharePoint groups are optional, meaning they may or may not need to be enrolled, depending on your requirements. For a table summarizing all enrolled entities, see Summary, Enrolling Control Center Entities.

Required Entities

Users and User Groups: To be able to create user components in Policy Studio, you use the Enrollment Manager utility to enroll users and user groups. With the Enrollment Manager, you have the choice of enrolling entities directly from an LDAP server (such as Microsoft Active Directory (AD)), groups from SharePoint, or information from any correctly formatted LDIF (Lightweight Directory Inter­change Format) file. You can enroll from more than one enrollment domain into the same Control Center. For users, hosts, and groups, you can also create Domain Groups to enroll information that spans multiple enrollment domains.

Hosts and Host Groups: To be able to create computer components, you must enroll hosts and host groups, also using Enrollment Manager. You can enroll them either from an LDAP server directory or from an LDIF file, and you can enroll from more than one AD domain. Hosts and Hosts Groups can also be enrolled through a Domain Group.

Applications: In order to be able to create Application Components, you must enroll any applications you will want to monitor. This topic will be covered in Enrolling Other Entities.

File Shares: You must enroll file share information to facilitate mapping to spe­cific resource paths. File shares are enrolled by querying the servers in your organization. To do this, you must manually run a utility called Resource Path Discovery. This topic will be covered in Enrolling Other Entities.

Optional Entities

SharePoint Groups: You will need to enroll SharePoint groups only if you have installed one or more SharePoint Enforcers, and plan to design policies to moni­tor and control those servers. (Note that SharePoint Enforcers are available only with the Entitlement Management product.) You enroll SharePoint groups with the Enrollment Manager. Details about SharePoint enrollment are covered in Enrolling Other Entities. SharePoint Groups can also be included in a Domain Group enrollment.

Location Sites: Location sites are physical or logical locations (such as Head­quarters, Boston Office, VPN), expressed as one or more ranges of IP addresses, that you can enroll and use to define location-based computer components. Site information is gathered from a text file you create manually, specifying the computers included in each site. You enroll location sites with a utility called Import Locations; this topic will be covered in the following chapter.

Windows and Linux

You enroll entities into your Control Center system the same way regardless of whether you will be installing enforcers on Windows file servers, Linux file serv­ers, or a mixture of both. (Note that file server enforcers are available with the Entitlement Management product only.) That is, there is no such thing as “Win­dows enrollment” or “Linux enrollment.” Rather, you just have to be sure your Samba server is configured so it can map the LDAP users in your Active Directory on the Windows side, to the users defined on the Linux side. Ordinarily this has to be done in any case, so the two systems can work together. As long as this is configured properly, Control Center will manage all enrolled entities internally, and policies will be enforced properly on both Linux and Windows servers.

The one exception to this is discovering and enrolling file shares, which requires that you run separate utilities on the Windows and Linux environments. For details, see About Resource Path Discovery.

About Domain Group Enrollment

A Domain Group contains multiple User, Host, and Group sources (referred to as “subdomains”), that can be managed together in a single enrollment process. The Domain Group option is required for enrollments where entities are located across multiple sources and there are interdependencies between the entities.

Consider the following example: users are stored in two different sources (one set of users in Active Directory and one set managed in an LDIF file). Both sets of users are referenced in Groups managed within SharePoint. To design policies around the SharePoint Groups, you would need to enroll all three sources into Control Center and retain information about Group membership across the two user domains (AD and LDIF). To do this, you would use the Domain Group enroll­ment option. You would list all pertinent enrollment sources as “subdomains” in the Domain Source file, and enroll and manage them as part of one process. For details on enrolling Domain Groups, see How to Enroll a Domain Group.

Enrollment Failures

Control Center considers each instance of enrolling a specified set of LDAP data as a distinct metadata entity called an enrollment. Information about enroll­ments is saved internally, independent of the actual data that was imported, and you can refer later to enrollments as you manage the system.

It is possible that an enrollment may fail for various reasons—a connectivity problem, improper syntax in a connection file or definition file, a typo in the command line parameters, and so on. Even if an enrollment attempt fails, it is still recorded as an enrollment, and is present in the management data reposi­tory. In such cases, once you discover and correct the problem that caused the failure, you must use the -delete command to manually delete the failed enroll­ment before you re-attempt it. For details on deleting enrollments, see Deleting Single Domain Enrollments  or Deleting Domain Group Enrollments.

For Domain Group enrollment, if a subdomain fails, all subsequent subdomains listed in the group will still be processed (synced or enrolled).

All events connected to enrollments are stored in a special log file; see Event Log Settings.
