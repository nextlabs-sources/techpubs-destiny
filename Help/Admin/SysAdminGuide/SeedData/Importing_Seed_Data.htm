Importing Seed Data

Seed Data are policy objects—action and application components, for example— that are necessary to write policies in Policy Studio. Before creating policies in Policy Studio, you must import Seed Data to support the NextLabs products in your implementation. Different NextLabs products require different Seed Data.

This section describes the following procedures:

·        Locating Seed Data Files

·        Configuring User Permissions

·        Running the createSeedData Tool

This chapter does not discuss how to configure custom obligations, which also must be configured before you create policies that require custom obligations. For more information about configuring obligations, consult the user guide for each Enforcer included in your implementation.

Configuring User Permissions

Only a specified group of users can import and use Seed Data.

·        To import Seed Data and make it available to other Policy Studio users, the user must belong to the All Policy Server Users group, and the user’s Default Access Control Group property must be set to All Policy Server Users.

·        To use Seed Data once it has been imported to Policy Studio, users must belong to the All Policy Server Users group. Users who are not part of this group see the Seed Data, but cannot use it.

Note: Do not use the Administrator user to import Seed Data. Doing so results in policies and components that can only be accessed and managed by Administrator.

For information about users and groups, see Managing Users and Managing User Groups.

This procedure assumes you have already created users who are allowed to use Policy Studio. For information about creating users, see Adding a New User.

The users added to the All Policy Server Users group now have access to the Seed Data in Policy Studio. The next step is to allow one of these users the ability to import Seed Data.

Designating a User to Import Seed Data

Once users have been assigned to the All Policy Server Users group, as described in the previous procedure, designate one or more users as the person who can import Seed Data that all users in the All Policy Server Users group can access. This person is typically assigned the System Administrator or Policy Administra­tor role and manages the life cycle of policies and policy components.

  1. In Administrator, under the Users and Roles tab, select Users.

  2. In the Users list, select the user that you want to designate as the person to import Seed Data.

  3. In the user definition, in Default Access Control Group, select All Policy Server Users, as shown in the following figure.

Running the createSeedData Tool

If you meet the requirements for importing Seed Data, as described in the previ­ous section, you can use the createSeedData.bat tool to import Seed Data into Policy Studio. This tool is located at the following location:

<install dir>\PolicyServer\tools\importexport 

Run this command for each Seed Data file you need to configure in Policy Studio. Components will be inserted into Policy Studio in Approved Status.

Follow these steps:

  1. Close Policy Studio, if it is open.

  2. Open a command prompt and change directories to <install dir>\
    PolicyServer\tools\importexport    .

  3. Enter the following command, supplying the appropriate arguments:

Note: The command is case sensitive.

createSeedData -s localhost -u <user> -w <password> -F <filename> -A -p <port>

Where:

·        -s: the host where the Policy Server is located. If this script is being run locally, enter localhost.

·        -u: the name of the user permitted to import Seed Data, as described in Designating a User to Import Seed Data. If the user name contains a space, enclose it in double quotation marks (" ").

·        -w: the user password.

·        -F (in upper case): the path to the Seed Data file you are attempting to import. Note that seed data is one level up, so, the file path should be written as follows:

../Seed_Data/Entitlement_Management-SAP-6.5.xml

·        -A (in upper case): this argument makes sure the file is inserted into Pol­icy Studio in Approved status. If you do not supply this argument, the com­ponents will be in Draft status.

·        -p (optional): the port of the Control Center host, which is 8443 unless it was explicitly changed. This argument is optional; if it is not present, the default value 8443 will be used.

Note: You can also run the command with -h (as the only argument) to dis­play definitions of all valid command line arguments.
